# 成熟游戏差距清单

> 这份不是 Steam 发售清单，而是“距离真正成熟、长期可维护、用户体验稳定的游戏”还差什么。
> 以当前项目状态为基准整理，尤其强化 UI 规范：全屏只能有一个主要 UI，只有弹出窗口允许压在主 UI 之上。

## 当前已确认的问题

- [ ] UI 还没有统一状态机，主菜单和世界地图都在直接 `SetActive(true/false)` 控多个面板。
- [ ] UI 规则还没有被代码强约束，目前靠调用顺序避免重叠，不够稳定。
- [ ] 还没有“主界面层 / 弹窗层 / Toast 层 / 遮罩层”这种明确分层。
- [ ] 主菜单虽然已有 `CloseAllPanels()` 思路，但仍然是局部约束，不是整个项目统一机制。
- [ ] 世界地图是手工关闭 `inventoryPanel / workshopPanel / sectPanel`，说明 UI 切换仍然是散点逻辑。
- [ ] 远征和世界地图大量 UI 由 runtime builder 直接拼装，后续一旦功能继续长，会更难控制层级、布局、复用和重叠。
- [ ] 还没有针对不同分辨率、比例、全屏模式做系统化 UI 验证，当前不能保证“不重叠”。

## [ ] A. UI 硬规范，必须先立

### UI-01. 全屏主界面只能同时存在一个

- [ ] 定义正式规则：任意时刻只能有一个 `FullScreen Main UI` 处于显示状态。
- [ ] 全屏主界面候选要明确列出来，例如：
- [ ] `MainMenu`
- [ ] `WorldMap`
- [ ] `Expedition`
- [ ] `BattleResult`
- [ ] 主界面之间只能“切换”，不能“叠加显示”。
- [ ] 禁止同时显示两个以上全屏面板，即使视觉上看起来没完全遮住也不允许。

### UI-02. 只有弹窗允许覆盖主界面

- [ ] 单独定义 `Modal Popup` 概念，和主界面彻底区分。
- [ ] 允许覆盖主界面的只包括：
- [ ] 设置弹窗
- [ ] 确认弹窗
- [ ] 奖励弹窗
- [ ] 教学提示弹窗
- [ ] 错误提示弹窗
- [ ] 弹窗必须带统一遮罩层，阻断下层点击。
- [ ] 弹窗关闭后必须回到原主界面，不能额外改乱 UI 状态。

### UI-03. 建立统一 UI 栈，而不是每个控制器各写一套

- [ ] 增加统一 `UIRoot` 或 `UIStateMachine`。
- [ ] 统一管理：
- [ ] 当前主界面是谁
- [ ] 当前是否存在弹窗
- [ ] 弹窗栈顺序
- [ ] 遮罩显示状态
- [ ] Esc / 返回键行为
- [ ] 主菜单、世界地图、远征界面都不能再直接各自散写 `SetActive` 流程。
- [ ] 以后新增 UI 时，必须先注册到 UI 系统，而不是再写一个自由漂浮的新 Panel。

### UI-04. 建立明确分层

- [ ] 至少固定 4 层：
- [ ] `MainLayer`：全屏唯一主界面
- [ ] `ModalLayer`：弹窗
- [ ] `BlockerLayer`：遮罩 / 点击拦截
- [ ] `ToastLayer`：短提示、飘字、非阻塞反馈
- [ ] 不允许把全屏面板和弹窗都挂在同一层级随便显隐。
- [ ] 每个层的排序规则要固定，不接受“看起来对就行”。

### UI-05. 从源头解决重叠问题

- [ ] 全屏 UI 禁止同时保留多个主要信息区争抢焦点。
- [ ] 一个页面只允许一个主任务区域，其他信息做辅助区，不要多个同等权重的大块内容同时争视线。
- [ ] 同一个区域里，文本、按钮、列表、详情不能互相压边或溢出。
- [ ] 需要建立最小分辨率基线，例如 `1920x1080` 和 `1280x720` 都不能重叠。
- [ ] 至少检查这些比例：
- [ ] `16:9`
- [ ] `16:10`
- [ ] `21:9`
- [ ] 窗口模式下缩放后仍不能出现主要按钮互相覆盖。

## [ ] B. 成熟 UI 还必须补齐的东西

### UI-06. 导航一致性

- [ ] 每个主界面都要有统一入口、统一返回路径、统一关闭逻辑。
- [ ] Esc 行为必须固定：
- [ ] 有弹窗时先关弹窗
- [ ] 无弹窗时执行当前主界面的返回逻辑
- [ ] 不允许有的界面 Esc 关窗口，有的界面没反应，有的界面直接退场景。
- [ ] 焦点切换逻辑要明确，后续做手柄时不能重写一遍 UI 架构。

### UI-07. 信息密度和阅读层次

- [ ] 每个页面先定义“玩家第一眼必须看到什么”，不要把系统信息同时堆满全屏。
- [ ] 标题、当前目标、核心资源、当前可执行动作要优先级明确。
- [ ] 长文本说明不要直接塞到主操作区。
- [ ] 帮助说明、背景故事、系统描述应进入二级信息层或弹窗。

### UI-08. 布局系统化

- [ ] 少写绝对坐标式的大块拼装，逐步收敛到可维护的布局规则。
- [ ] 列表、按钮组、详情区、标题区要有统一间距规范。
- [ ] 字号、边距、标题层级、按钮尺寸要形成 design token 或常量定义。
- [ ] 不能继续依赖“这个 prefab 目前看起来没问题”作为布局标准。

### UI-09. 视觉反馈完整

- [ ] 所有可点击元素都需要悬停、按下、不可用三态。
- [ ] 所有关键操作都需要明确反馈：
- [ ] 成功
- [ ] 失败
- [ ] 资源不足
- [ ] 已保存
- [ ] 进入新界面
- [ ] 不能让玩家点完按钮后猜系统有没有响应。

## [ ] C. 游戏体验距离成熟还差的内容

### UX-01. 新手引导

- [ ] 玩家第一次启动时必须知道：
- [ ] 我是谁
- [ ] 现在能做什么
- [ ] 先点哪里
- [ ] 为什么要继续玩
- [ ] 主菜单 -> 世界地图 -> 历练 -> 结算 这条路径要能被第一次游玩的玩家独立走通。

### UX-02. 主循环表达要清晰

- [ ] 当前项目已经有主菜单、世界地图、远征三段，但还需要把“为什么来回切换”讲清楚。
- [ ] 每轮行动都要有目标、成本、风险、收益，不然只是切场景，不是循环。
- [ ] 玩家应该始终知道当前最值得做的下一件事。

### UX-03. 设置与基础交互

- [ ] 当前只有基础音量和全屏还不够。
- [ ] 还需要至少补齐：
- [ ] 分辨率
- [ ] 窗口模式
- [ ] 画质基础项
- [ ] 输入说明
- [ ] 音频分组音量
- [ ] 退出确认或自动保存提示

### UX-04. 失败恢复和异常体验

- [ ] 存档损坏、加载失败、资源缺失、场景缺失都要有可理解提示。
- [ ] 不允许用控制台报错替代玩家提示。
- [ ] 玩家任何时候遇到错误，都应该知道：
- [ ] 发生了什么
- [ ] 现在还能不能继续
- [ ] 要不要重试或返回主界面

## [ ] D. 内容成熟度

### CONTENT-01. 核心内容量还要补

- [ ] 主玩法至少要能支撑玩家连续玩 30 分钟不靠脑补。
- [ ] 不同流派、地区、事件、敌人、奖励要开始形成差异，而不是同结构换文案。
- [ ] 世界地图、门派驻地、远征内容之间要有持续互相影响。

### CONTENT-02. 数值和节奏

- [ ] 成长曲线要明确，不要只靠后面再调。
- [ ] 玩家前 10 分钟、30 分钟、1 小时分别应该获得什么体验，需要定稿。
- [ ] 奖励和消耗不能失衡，否则任何 UI 再好看也撑不住长期体验。

### CONTENT-03. 长期目标

- [ ] 成熟游戏不能只有短期操作，还要有中期和长期目标。
- [ ] 玩家要知道自己为什么继续推进：
- [ ] 解锁什么
- [ ] 变强什么
- [ ] 改变世界什么
- [ ] 获得什么身份或结局

## [ ] E. 技术成熟度

### TECH-01. UI 代码要从“能跑”升级为“可维护”

- [ ] 减少超长 runtime builder 持续膨胀。
- [ ] 把界面结构、组件引用、状态切换拆成清晰职责。
- [ ] 逐步避免把布局、文案、逻辑、状态更新全塞进一个类。
- [ ] UI 系统层要先定，再继续堆功能，不然之后每加一屏都会更乱。

### TECH-02. 自动化验证

- [ ] 至少补一轮 UI 状态测试：
- [ ] 打开主菜单时没有其他主界面激活
- [ ] 打开设置时主界面未重复实例化
- [ ] 世界地图打开背包时其他主面板被关闭
- [ ] 弹窗关闭后回到正确界面
- [ ] 场景切换后不会残留旧 UI
- [ ] 至少补一轮分辨率冒烟检查，验证主要页面不重叠。

### TECH-03. 资源和加载

- [ ] 统一 UI 资源加载入口，减少散落的 `Resources.Load`。
- [ ] 明确哪些 UI 用 prefab，哪些允许 runtime 生成。
- [ ] 不要让相同职责的界面同时存在 prefab 版和 runtime 版长期并行。

## [ ] F. 建议先做的顺序

1. [ ] 先定 UI 铁规则：一个主界面 + 弹窗层，不允许主界面并列重叠。
2. [ ] 建统一 UI 状态机，把主菜单、世界地图、远征都纳入同一套切换规则。
3. [ ] 跑一轮分辨率和全屏/窗口模式检查，把可见重叠全部清掉。
4. [ ] 再补设置、引导、错误提示、状态反馈，让基础体验过线。
5. [ ] 最后继续补内容量、节奏、长期目标和技术验证。

## 现阶段结论

- [ ] 当前项目已经不是“什么都没有”，但距离“成熟游戏”差的主要不是再多几个系统，而是缺少统一产品规范，尤其是 UI 规范和交互稳定性。
- [ ] 你提的“全屏只能有一个主要 UI，除非是弹出窗口”是对的，而且应该直接升格为项目级硬约束。
- [ ] 如果后续继续推进，最值得优先落地的不是再加新页面，而是先把 UI 框架定死，再逐个页面按规则整改。
