# Steam 上线检查清单

> 面向当前项目实际状态整理，不做“兼容旧版本”前提，按“要能卖、能过审、能稳定更新”来倒推。

## 当前已确认的缺口

- [ ] 还没有接入 Steamworks：项目内未发现 `SteamAPI`、`SteamManager`、成就、统计、云存档相关实现。
- [ ] 还没有本地化框架：当前未发现正式的多语言系统。
- [ ] 项目基础信息还是占位值：`companyName = DefaultCompany`、`productName = TutorialGame`、`bundleVersion = 1.0`。
- [ ] 当前构建场景只有 3 个：`Main`、`WorldMap`、`Game`，还不能说明完整上线流程已经打通。
- [ ] 现有存档系统还需要按“正式发售标准”补稳定性验证、损坏恢复、升级策略。

## [ ] 必须完成

### 1. 游戏本体必须形成可售卖闭环

- [ ] 明确玩家主循环：开始游戏 -> 学习规则 -> 成长推进 -> 遭遇挑战 -> 获得反馈 -> 继续投入。
- [ ] 明确失败与恢复逻辑：失败后是否惩罚、如何重开、是否会卡死进度。
- [ ] 主线目标必须可描述：玩家为什么继续玩、玩到什么时候算一轮完成。
- [ ] 至少做出一个“完整可体验版本”，而不是几个场景拼接。
- [ ] 新玩家从启动到第一次“理解玩法”不应依赖开发者说明。
- [ ] 需要有最小内容量标准：
  - [ ] 可重复玩的核心机制已经稳定。
  - [ ] 至少有基础数值成长、资源消耗、奖励回路。
  - [ ] 至少有能支撑商店截图和 30 分钟体验的视频内容。

### 2. 存档系统必须达到正式发售标准

- [ ] 定义唯一正式存档结构，不再继续保留过渡方案。
- [ ] 明确每次保存的触发点：进入场景、退出场景、关键行为、退出游戏。
- [ ] 实现存档损坏处理：
  - [ ] 坏档时不能直接卡死或黑屏。
  - [ ] 至少要能回退到默认档或备份档。
- [ ] 实现原子写入或临时文件替换，避免中途断电导致坏档。
- [ ] 明确版本升级策略：
  - [ ] 发售前可以不兼容旧开发档。
  - [ ] 但发售后版本之间必须定义升级逻辑或明确删档策略。
- [ ] 验证完整路径：
  - [ ] 首次启动无存档。
  - [ ] 有存档继续游戏。
  - [ ] 场景切换后重进。
  - [ ] 异常退出后重开。
  - [ ] 坏档恢复。

### 3. 发布构建必须可交付

- [ ] 先只收敛 Windows 正式版本，别同时摊多平台。
- [ ] 统一产品标识：
  - [ ] 正式公司名。
  - [ ] 正式游戏名。
  - [ ] 正式版本号规范。
- [ ] 清理开发期开关：
  - [ ] 调试按钮。
  - [ ] 测试入口。
  - [ ] 作弊逻辑。
  - [ ] 仅编辑器可用的假数据路径。
- [ ] 明确启动行为：
  - [ ] 是否全屏。
  - [ ] 默认分辨率。
  - [ ] 是否支持窗口模式切换。
  - [ ] 首次启动配置落地。
- [ ] 明确退出流程：主菜单退出、游戏中退出、Alt+F4、崩溃后重开。
- [ ] 构建产物目录、命名、版本号、提交记录要有固定规则。
- [ ] 发售包里不能混入开发无关资源、测试图、废弃脚本依赖。

### 4. 基础体验必须补齐

- [ ] 设置菜单必须完整可用：
  - [ ] 音量。
  - [ ] 分辨率/窗口模式。
  - [ ] 输入相关配置或至少展示按键说明。
- [ ] 新手引导必须存在，哪怕是极简版本。
- [ ] UI 可读性必须过线：
  - [ ] 小字是否能看清。
  - [ ] 高对比信息是否清晰。
  - [ ] 色彩是否影响关键反馈识别。
- [ ] 音频反馈要成体系：点击、确认、失败、奖励、战斗/事件反馈。
- [ ] 加载与切场不能显得像程序异常：
  - [ ] 黑屏时间可接受。
  - [ ] 有必要时加 loading 表现。

### 5. 性能和稳定性必须达标

- [ ] 建立最低性能基线：目标机器、目标分辨率、目标帧率。
- [ ] 检查主菜单、世界地图、主玩法场景的 CPU/GPU/内存波动。
- [ ] 长时间运行测试：
  - [ ] 连续玩 30 分钟。
  - [ ] 连续玩 1 小时。
  - [ ] 多次切场景后不明显涨内存。
- [ ] 修掉会影响普通玩家的错误：
  - [ ] 红字异常。
  - [ ] 空引用。
  - [ ] 丢引用。
  - [ ] 首次进入卡死。
- [ ] 必须能稳定从全新安装包启动到进入可玩状态。

### 6. Steamworks 接入必须做实

- [ ] 接入 Steamworks SDK 或可靠封装库。
- [ ] 打通最小启动链路：
  - [ ] Steam 启动游戏。
  - [ ] Overlay 可正常呼出。
  - [ ] AppID 配置正确。
- [ ] 明确是否做以下功能，并一次性定版：
  - [ ] 成就。
  - [ ] 统计。
  - [ ] 云存档。
  - [ ] Rich Presence。
- [ ] 如果做成就：
  - [ ] 条件必须能在代码中稳定触发。
  - [ ] 不能出现误发、漏发、重复发逻辑问题。
- [ ] 如果做云存档：
  - [ ] 先确认本地存档路径和文件组织。
  - [ ] 再确认 Steam Cloud 映射规则。
- [ ] 配置 Steam 分支与上传流程：
  - [ ] 内测分支。
  - [ ] 候选发布分支。
  - [ ] 正式分支。

### 7. 商店页与法务素材必须补齐

- [ ] 商店页素材：
  - [ ] Capsule 图。
  - [ ] Header 图。
  - [ ] 游戏截图。
  - [ ] Trailer。
- [ ] 商店文案：
  - [ ] 一句话卖点。
  - [ ] 短描述。
  - [ ] 长描述。
  - [ ] 标签。
- [ ] 年龄和内容披露按 Steam 要求准备。
- [ ] 第三方资产授权要清点完：
  - [ ] 插件。
  - [ ] 字体。
  - [ ] 音乐。
  - [ ] 音效。
  - [ ] 美术素材。
- [ ] 必须确认是否需要隐私政策、EULA、第三方声明页。

## [-] 建议尽快完成

### 8. 本地化与全球发行准备

- [ ] 如果目标不是只卖中文区，至少补中英双语。
- [ ] 文案不要散落硬编码，先收敛到统一出口。
- [ ] UI 需要预留文本长度变化空间。
- [ ] 数字、时间、货币、输入提示不要写死。

### 9. QA 与回归体系

- [ ] 建一个最小人工回归清单：
  - [ ] 新游戏。
  - [ ] 继续游戏。
  - [ ] 存档/读档。
  - [ ] 场景切换。
  - [ ] 设置项保存。
  - [ ] 退出重开。
- [ ] 为核心逻辑补 EditMode 测试，尤其是：
  - [ ] 存档读写。
  - [ ] 关键数值计算。
  - [ ] 奖励/结算逻辑。
- [ ] 每次准备打包前都跑一轮固定回归。

### 10. 数据、平衡和节奏

- [ ] 核心资源产出和消耗曲线要能说清楚。
- [ ] 不能靠“以后再调”掩盖当前节奏问题。
- [ ] 新玩家前 10 分钟、30 分钟、1 小时体验节奏要分别验证。
- [ ] 至少做一轮外部试玩，不要只靠开发者自己判断。

### 11. Steam Deck / 手柄支持

- [ ] 如果准备支持 Steam Deck，就必须单独做一轮适配验证。
- [ ] 如果准备支持手柄，就补齐：
  - [ ] 输入映射。
  - [ ] UI 焦点导航。
  - [ ] 提示图标切换。
- [ ] 如果暂时不支持，就不要在商店页误导用户。

## [ ] 上线前最终确认

### 12. 发售前一周冻结项

- [ ] 冻结存档结构，避免发售前再改档格式。
- [ ] 冻结成就 ID、统计项 ID、云存档路径。
- [ ] 除 P1 问题外，不再继续做高风险重构。
- [ ] 每个候选构建都保留版本号、构建时间、对应提交号。

### 13. 最终发布演练

- [ ] 从零环境安装 Steam 构建包。
- [ ] 实测首次启动、第二次启动、断网启动、异常退出后重开。
- [ ] 实测商店页跳转、Overlay、成就、云存档。
- [ ] 实测卸载重装后的用户体验。
- [ ] 至少让 1 个不了解项目的人完整玩 30 分钟并记录问题。

### 14. 发布日准备

- [ ] 发布当天负责人分工明确：谁看商店页、谁看构建、谁看玩家反馈。
- [ ] 准备首日热修复流程。
- [ ] 准备已知问题清单，避免临场混乱。
- [ ] 准备客服/社区回应模板，至少覆盖存档、启动失败、设置异常三类问题。

## 建议执行顺序

1. [ ] 先把“游戏闭环 + 存档稳定 + 正式构建”打通。
2. [ ] 再接 Steamworks 最小链路，先跑通启动、Overlay、上传分支。
3. [ ] 然后补商店页素材、成就、云存档、本地化。
4. [ ] 最后做性能、回归、发布演练和上线冻结。

## 现阶段结论

- [ ] 当前项目距离“可以放到 Steam 售卖”还有明显距离，主要缺口不是单点代码，而是“产品闭环、发布工程、Steam 接入、商店素材、QA 流程”都还没封口。
- [ ] 如果要继续推进，建议后续把这份清单拆成实际执行任务，按模块逐项落地并持续勾选。
